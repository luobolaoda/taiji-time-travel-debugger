## 1. 概述

**太极时光倒流调试器** 是一个革命性的 C/C++ 调试工具。与传统的"断点→单步→观察"调试方式不同，它实现了 **二进制级别的时光倒流** ——程序崩溃后，你可以像倒放录像带一样，逐条回退程序的写内存操作，精确找到哪条指令破坏了数据。

### 核心理念

```
传统调试:  向前执行 → 到达断点 → 查看状态
时光倒流:  执行到崩溃 → 回溯历史 → 查看任意时刻状态
```

这来自太极指令集架构中 **阳指令的逆执机制**：每条写操作在写入新值前，先保存旧值。崩溃后逆序恢复旧值，即可"倒流"到之前任意时刻。

### 特性

| 特性 | 说明 |
|------|------|
| 环形缓冲区 | 默认 1024 条快照，崩溃时保留最近操作 |
| 自动检查点 | 缓冲区满时自动生成全量检查点 |
| 交互式调试 | 崩溃时进入命令行，支持回溯/查看/恢复 |
| 非交互模式 | 设置环境变量后自动打印快照摘要退出 |
| 跨平台 | Windows (MinGW/MSVC) + Linux + macOS |
| 零依赖 | 仅标准 C11 库，无外部依赖 |

## 2. 快速开始

### 环境要求

- GCC ≥ 4.9 或 Clang ≥ 3.5 或 MSVC ≥ 2015
- Windows / Linux / macOS
- （可选）LLVM ≥ 15 用于编译器自动插桩

### 编译

```bash
# 编译运行时库
gcc -std=c11 -O0 -g -c snapshot_runtime.c -o snapshot_runtime.o

# 编译演示程序
gcc -std=c11 -O0 -g example/crash_demo.c snapshot_runtime.o -o crash_demo

# 编译命令行调试器
gcc -std=c11 -O0 -g taiji-debug.c snapshot_runtime.o -o taiji-debug
```

或一键：

```bash
# Windows
test_all.bat

# Linux/macOS
chmod +x build.sh && ./build.sh
```

### 运行演示

```bash
# 数据损坏追踪（安全场景，推荐首个尝试）
./crash_demo 6

# 多步操作 + 检查点
./crash_demo 7

# 全部安全场景
./crash_demo 0

# 崩溃场景（非交互模式，自动打印快照）
TAIJI_NONINTERACTIVE=1 ./crash_demo 3
```

## 3. 核心概念

### 3.1 快照（Snapshot）

每次写内存操作前，运行时自动记录一条快照：

```
快照 = { 地址, 旧值, PC, 大小 }
```

- **地址**：被修改的内存地址
- **旧值**：写入前的值（8 字节）
- **PC**：触发写入的指令地址
- **大小**：写入字节数（1/2/4/8）

### 3.2 环形缓冲区（Ring Buffer）

快照存储在固定大小的环形缓冲区中（默认 1024 条）。新快照覆盖最旧的快照，确保崩溃时总能保留最近的操作历史。

```
[旧快照] → [旧快照] → ... → [新快照] → [最新快照]
                                ↑ 写指针
```

### 3.3 检查点（Checkpoint）

当环形缓冲区接近满时，自动创建一个全量检查点，保存当前寄存器状态和栈帧快照。回溯到检查点可以一次性跳过大量操作。

### 3.4 逆执（Reverse Execution）

回溯过程：从环形缓冲区末尾开始，逐条读取快照，将旧值写回内存地址。

```
快照[最新]: 地址=0x1234, 旧值=42 → 写回 0x1234 = 42
快照[上条]: 地址=0x5678, 旧值=99 → 写回 0x5678 = 99
...
```

### 3.5 两种使用模式

| 模式 | 适用场景 | 快照来源 |
|------|----------|----------|
| **手动模式**（本教程演示） | 学习/原型/无 LLVM 环境 | 手动调用 `SIMULATE_SNAPSHOT` 宏 |
| **自动模式**（生产环境） | 实际项目调试 | LLVM Pass 自动在每条 store 前插入 `snapshot_record()` |

## 4. 三步上手教程

### 第一步：理解数据损坏问题

```c
int data = 42;

data = 100;    // 正常
data = 200;    // 正常
data = -1;     // ← 意外写入！谁干的？
data = 300;    // 正常
```

没有快照时，你只能看到最终值 300，无法知道中间被写成了 -1。

### 第二步：用快照追踪

运行场景 6：

```bash
$ ./crash_demo 6

══ 场景6: 数据损坏追踪 ══
  初始值: g_corrupt_data = 42
  写入 100: g_corrupt_data = 100
  写入 200: g_corrupt_data = 200
  意外写入 -1: g_corrupt_data = -1      ← 可疑！
  写入 300: g_corrupt_data = 300

  程序正常退出
  最终快照数: 4                        ← 4 次写入各记录 1 条
```

## 5. 崩溃演示场景详解

`crash_demo` 程序包含 7 个场景（0-7），涵盖常见的崩溃类型：

| 场景 | 名称 | 是否崩溃 | 用途 |
|------|------|----------|------|
| 0 | 全部安全场景 | 否 | 运行场景 6+7，展示完整流程 |
| 1 | 数组越界写入 | 否（Windows 上通常不崩） | 演示内存越界但不崩溃的情况 |
| 2 | 释放后使用 (UAF) | 是 | 演示 free 后继续写入导致崩溃 |
| 3 | 空指针解引用 | 是 | 经典 `*NULL = 42` 崩溃 |
| 4 | 栈溢出 | 是 | 无限递归耗尽栈空间 |
| 5 | 除零 | 是 | 整数除零触发 SIGFPE |
| 6 | 数据损坏追踪 | 否 | 多次写入中间值异常，用快照追踪 |
| 7 | 多步操作 + 检查点 | 否 | 10 步累加，每 5 步生成检查点 |

### 场景 2：释放后使用

```c
char* buf = malloc(256);
strcpy(buf, "Hello, Taiji Snapshot!");
free(buf);                          // ← 释放
strcpy(buf, "CORRUPTED!");          // ← 崩溃点：写入已释放内存
```

崩溃时快照会保存 `free` 前后的 `buf` 指针值和内容，回溯可以看到释放前缓冲区里的数据。

### 场景 4：栈溢出

```c
void recursive_overflow(int depth) {
    char local[1024];               // 每次 ~1KB 栈消耗
    recursive_overflow(depth + 1);  // 无限递归
}
```

崩溃时，快照会记录每次递归调用前的栈状态。可以回溯查看是在哪一层递归时崩溃的。

## 6. 交互式调试命令

崩溃时自动进入交互模式，可用命令：

| 命令 | 简写 | 说明 | 示例 |
|------|------|------|------|
| `w N` | | 回溯 N 步 | `w 10` |
| `wc N` | | 恢复到检查点 N | `wc 3`，`wc`（最近） |
| `s` | | 显示最近 20 条快照 | `s` |
| `c` | | 显示所有检查点 | `c` |
| `i` | `info` | 显示系统状态 | `i` |
| `p` | | 暂停/恢复快照记录 | `p` |
| `m ADDR` | `mem` | 读取内存地址 | `m 0x7fff1234` |
| `h` | `help` | 显示帮助 | `h` |
| `q` | `quit` | 退出调试器 | `q` |

### 非交互模式

设置环境变量 `TAIJI_NONINTERACTIVE=1` 后，崩溃时自动打印快照摘要并退出，不等待用户输入。适用于 CI/CD 或自动化测试：

```bash
TAIJI_NONINTERACTIVE=1 ./crash_demo 3
```

## 7. LLVM Pass 集成（生产环境）

以上演示使用手动 `SIMULATE_SNAPSHOT` 宏。在生产环境中，应使用 LLVM Pass 自动插桩：

### 原理

```
源代码:
  g_array[i] = i * 100;

LLVM IR 插桩后:
  __taiji_snapshot_store(&g_array[i], g_array[i], __builtin_return_address(0), 4);
  g_array[i] = i * 100;
```

### 编译步骤

```bash
# 1. 编译 Pass
cd llvm-pass
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make TaijiSnapshotPass

# 2. 编译目标程序（带插桩）
clang -fpass-plugin=./build/TaijiSnapshotPass.so \
      -g -O0 \
      my_program.c snapshot_runtime.c \
      -o my_program

# 3. 运行
./my_program
```

### 性能影响

| 模式 | 性能开销 | 适用场景 |
|------|----------|----------|
| 全量快照（每条 store） | ~10x 慢 | 开发/调试 |
| 采样快照（每 N 条 store） | ~2-3x 慢 | 性能测试 |
| 仅关键变量 | ~1.2x 慢 | 生产监控 |

## 8. 架构原理

### 系统架构

```
┌─────────────────────────────────────────────────┐
│                   用户程序                        │
│  ┌──────────────────────────────────────────┐   │
│  │  store 指令 → snapshot_record() 插桩     │   │
│  │  main() → snapshot_init()                │   │
│  │  main() → crash_handler_install()        │   │
│  └──────────────────────────────────────────┘   │
│                      │                           │
│                      ▼                           │
│  ┌──────────────────────────────────────────┐   │
│  │          snapshot_runtime                │   │
│  │  ┌──────────┐  ┌──────────┐             │   │
│  │  │ 环形缓冲区 │  │ 检查点表  │             │   │
│  │  │ (1024条) │  │  (64个)  │             │   │
│  │  └──────────┘  └──────────┘             │   │
│  │         │            │                    │   │
│  │         ▼            ▼                    │   │
│  │  ┌──────────────────────────┐            │   │
│  │  │   交互式调试引擎          │            │   │
│  │  │   回溯/查看/恢复/检查点   │            │   │
│  │  └──────────────────────────┘            │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 与太极 ISA 的对应关系

| 太极 ISA 概念 | 时光倒流实现 |
|---------------|-------------|
| 阳指令（执行前保存旧值） | `snapshot_record()` 在 store 前保存 |
| 逆执（逐条恢复旧值） | `snapshot_rewind()` 从缓冲区弹出 |
| 检查点 | `snapshot_create_checkpoint()` |
| 异常处理 | `crash_handler_install()` 捕获 SIGSEGV/SIGFPE 等 |

### 线程安全

- 环形缓冲区操作使用 `CRITICAL_SECTION`（Windows）或 `pthread_mutex`（POSIX）
- 快照计数器使用 C11 `atomic_int`
- 检查点表使用独立锁，避免与快照记录竞争

## 9. API 参考

### 初始化与清理

```c
void snapshot_init(void);           // 初始化快照系统（main 开头调用）
void snapshot_reset(void);          // 重置所有快照数据
```

### 快照记录

```c
void snapshot_record(void* addr, uint64_t old_value, uint64_t pc, int size);
// addr:      被修改的内存地址
// old_value: 写入前的旧值
// pc:        指令地址（__builtin_return_address(0)）
// size:      写入大小（1/2/4/8）
```

### 回溯操作

```c
int  snapshot_rewind(int steps);                    // 回溯 N 步，返回实际步数
bool snapshot_rewind_to_checkpoint(int cp_index);   // 回溯到检查点，-1=最近
```

### 查询接口

```c
int    snapshot_count(void);                        // 当前快照数
int    snapshot_checkpoint_count(void);             // 当前检查点数
double snapshot_buffer_usage(void);                 // 缓冲区使用率 (0.0~1.0)
const char* snapshot_debug_info(void);              // 调试信息字符串
```

### 列表查询（调试器用）

```c
int snapshot_list_snapshots(SnapshotSummary* out, int max);
int snapshot_list_checkpoints(CheckpointSummary* out, int max);
```

### 控制

```c
void snapshot_pause(void);      // 暂停快照记录
void snapshot_resume(void);     // 恢复快照记录
bool snapshot_is_paused(void);  // 是否已暂停
```

### 崩溃处理

```c
void crash_handler_install(void);    // 安装信号处理器
void crash_handler_uninstall(void);  // 卸载信号处理器
```

## 10. 常见问题

### Q: 为什么快照数为 0？

A: 快照需要手动插入或 LLVM Pass 自动插入。在演示程序中，已通过 `SIMULATE_SNAPSHOT` 宏在每次写操作前手动插入。

### Q: 崩溃时程序卡住不动？

A: 崩溃后进入交互调试模式，等待用户输入命令。输入 `q` 退出，或设置 `TAIJI_NONINTERACTIVE=1` 自动退出。

### Q: 回溯后程序能继续执行吗？

A: 回溯只恢复内存数据，不能恢复指令指针。如需真正"继续执行"，需要配合 LLVM Pass 保存完整的寄存器上下文。

### Q: 环形缓冲区满了会怎样？

A: 自动创建检查点，然后新快照覆盖最旧快照。检查点保存了完整的寄存器/栈帧快照。

### Q: 多线程程序能用吗？

A: 可以。快照记录使用原子操作和临界区保护，线程安全。但回溯时各线程的内存修改都会回退。

### Q: 性能开销多大？

A: 手动模式几乎无开销。LLVM Pass 全量插桩约 10x 慢（仅开发调试用），采样模式约 2-3x。

### Q: 支持哪些崩溃类型？

A: SIGSEGV（段错误）、SIGFPE（除零/浮点异常）、SIGILL（非法指令）、SIGABRT（断言失败）。Windows 上对应 ACCESS_VIOLATION、INT_DIVIDE_BY_ZERO、ILLEGAL_INSTRUCTION、STACK_OVERFLOW。
